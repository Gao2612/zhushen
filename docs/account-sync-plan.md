# 登录 / 账号 / 联网同步方案

本文记录未来实现联网账号与多设备同步时的方案。当前阶段不实现真正登录，只先落地本地用户档案，避免过早引入后端、鉴权和数据安全复杂度。

## 当前阶段：本地用户档案

目标：让用户拥有“这是我的空间”的体验，同时保持低风险、低维护成本。

已规划的数据结构：

- `profileId`：本机档案唯一 ID。
- `nickname`：用户昵称。
- `avatar`：本地头像标识，当前先用 1-2 个字展示。
- `bio`：备注签名。
- `preferences`：偏好设置，例如背景音乐、减少动态效果、记住上次浏览位置。
- `favorites`：收藏内容预留。
- `history`：浏览记录预留。
- `cloudBackup`：云备份状态预留。
- `createdAt` / `updatedAt`：创建与更新时间。

本地档案存储在 Electron `userData` 目录下的 `profile.json`。支持导入 / 导出 JSON，便于用户手动备份，也为未来云同步定义数据边界。

## 未来阶段：真正登录系统

只有在明确需要多设备同步、云端存档、账号找回或社交关系后，再进入该阶段。

需要实现：

- 注册、登录、退出。
- 第三方登录或验证码登录。
- 用户数据库。
- token 鉴权与刷新。
- 云端档案存储。
- 多设备同步。
- 手机端和桌面端数据冲突处理。
- 隐私政策、数据删除、账号找回。
- 后端部署、监控、备份和维护。

## 推荐后端结构

最小可行结构：

- `users`：账号基础信息。
- `profiles`：用户档案。
- `profile_snapshots`：导入 / 导出或自动备份快照。
- `devices`：设备标识。
- `sync_events`：同步日志。

档案同步建议采用“服务端记录最后更新时间 + 客户端保留本地更新时间”的方式。冲突时先不自动覆盖，提示用户选择“保留本机 / 使用云端 / 导出备份后合并”。

## 风险

- 账号系统会引入隐私和安全责任。
- 后端服务一旦上线，就需要持续维护。
- 云同步如果没有冲突策略，容易覆盖用户本地数据。
- 第三方登录或验证码会带来额外平台成本。

## 当前结论

现阶段只做本地档案和导入 / 导出。联网登录与云同步先保留入口和方案文档，等产品需求明确后再实现。
